System Overview

This system predicts the probability that the home team wins each MLB regular season game, covering seasons 2000 through 2026. It combines 25 years of historical data from multiple authoritative sources, engineers over 100 pre-game features, and trains six different machine learning models. The default production model is a stacked ensemble that blends all five base models for optimal accuracy and calibration.

Key design principles: All metrics are fully out-of-sample (no future data ever leaks into training). Prediction snapshots are immutable. Every derived artifact is traceable back to its source data. The system is designed for long-term sabermetric research.

The end-to-end pipeline flows through four stages:

  1. Data ingestion — fetch raw data from 6 external sources into structured Parquet files
  2. Feature engineering — compute 100+ pre-game features per game from the raw data
  3. Model training — train 6 models with time-weighted samples, calibration, and hyperparameter optimisation
  4. Prediction & monitoring — generate immutable prediction snapshots and track model drift over time

Baseball Statistics Primer

Understanding the model's features requires familiarity with the baseball statistics they encode. This section explains the key metrics used throughout the system, from traditional box-score stats to modern sabermetric measures.

Batting Statistics

Barrel%Barrel RatePercentage of batted balls with the ideal exit velocity (≥98 mph) and launch angle (26–30°) for extra-base hits. A proxy for consistent hard, productive contact.
Hard Hit%Hard Hit RatePercentage of batted balls with exit velocity ≥ 95 mph. Broader than barrel rate, captures overall contact quality.

Advanced / Sabermetric Measures

StatNameWhat It Measures
EloElo RatingA chess-style rating system adapted for baseball. Every team starts at 1500. After each game, the winner gains points and the loser drops points, with the magnitude depending on the upset factor. Accounts for opponent quality over time.
PythagoreanPythagorean ExpectationEstimates a team's expected winning percentage from runs scored (RS) and runs allowed (RA): RS² / (RS² + RA²). Teams that win more than their Pythagorean expectation are considered "lucky" and likely to regress.

Data Sources & Ingestion

The system ingests data from six external sources. Each source feeds into the feature engineering pipeline to construct a rich pre-game profile for every matchup.

MLB Stats API

The official MLB Stats API (statsapi.mlb.com) provides the game schedule and individual pitcher stats. Our async client uses a token-bucket rate limiter (5 requests/sec, burst of 10) with bounded concurrency, exponential backoff, and 429 (rate-limit) handling. All raw JSON responses are cached locally so repeat runs never hit the API.

  • Schedule data: Every regular season game from 2000–2026 with game IDs, teams, venue, date/time, doubleheader flags
  • Pitcher stats: Season-level ERA, K/9, BB/9, WHIP for every pitcher — used for prior-season starting pitcher quality features

Retrosheet Game Logs

Retrosheet is the gold standard for historical baseball data. Each game log contains the final score, starting pitchers, lineups (when available), and dozens of box-score fields. The system downloads game log text files from the Chadwick GitHub mirror (primary) or the Retrosheet.org ZIP archive (fallback), parses them into structured DataFrames, and stores them as Parquet files.

Retrosheet data powers the core rolling stats, Elo ratings, pitcher ERA calculations, and park factors. It is the backbone of all historical feature engineering.

FanGraphs

FanGraphs provides team-level advanced metrics: wOBA, Barrel%, Hard Hit%, ISO, BABIP, xwOBA (batting) and FIP, xFIP, K%, BB%, HR/FB, WHIP (pitching). These are fetched via the pybaseball library and joined to each game as prior-season team strength indicators. Available from 2002 onward.

Statcast (Baseball Savant)

Statcast is MLB's ball-tracking system, measuring exit velocity, launch angle, sprint speed, and spin rate since 2015. The system fetches individual batter and pitcher Statcast data via pybaseball and computes:

  • Lineup-weighted xwOBA — the average expected wOBA across a team's 9-man batting order
  • Lineup-weighted Barrel% — the average barrel rate across the lineup
  • Starting pitcher xwOBA allowed — how hard the opposing starter was hit, per Statcast

Player IDs are mapped from Retrosheet to MLBAM IDs using the Chadwick Baseball Bureau register.

Vegas Odds

Pre-game money-line odds are converted to implied win probabilities using the formula: 

If ML < 0 (favourite):  prob = |ML| / (|ML| + 100)
If ML > 0 (underdog):   prob = 100 / (ML + 100)

The vig (bookmaker margin) is removed by normalising so the two sides sum to 1.0. Vegas lines encode public money, sharp action, injury news, and weather — information that statistical models often miss. When available, the line movement (opening line vs. closing line) is also captured as a feature.

Weather

Historical weather data is fetched from the Open-Meteo API (free, no API key required) using the GPS coordinates of each ballpark. Three features are extracted: temperature (°F), wind speed (mph), and humidity (%). Extreme heat and wind can measurably affect batting outcomes.

Crosswalk (Retrosheet ↔ MLB API)

Retrosheet and the MLB Stats API use different game identifiers. The crosswalk matches them by (date, home team, away team, doubleheader number), producing a game_id_map_{season}.parquet with match confidence scores. The system enforces a minimum 99% coverage threshold — if fewer than 99% of games are matched, the crosswalk reports an error.

Feature Engineering

Every game is described by 100+ pre-game features computed from the raw data sources. Features are always computed before the game occurs — no future information ever leaks into the feature set. Cross-season continuity is maintained: rolling windows and Elo ratings carry forward from the previous season.

Elo Ratings (3 features)

The system computes pre-game Elo ratings for every team using a sequential update rule. After each game, the winner gains points and the loser drops them, with the magnitude proportional to how unexpected the outcome was: 

Expected = 1 / (1 + 10^((OpponentElo - TeamElo) / 400))
NewElo = OldElo + K × (Actual - Expected)
  • K-factor = 20 — the maximum rating swing per game
  • Home field advantage = 24 Elo points — the home team's rating is boosted by 24 before computing Expected
  • Season regression = 33% — at each season start, ratings regress 1/3 toward 1500 to account for roster turnover

The features are: home_elo, away_elo, and elo_diff (home minus away). An Elo difference of +100 corresponds roughly to a 64% expected win rate.

Rolling Team Statistics (45+ features)

Multi-window rolling statistics capture team form at different time scales. For each of five windows (7, 14, 15, 30, and 60 games), the system computes:

StatFeaturesDescription
Win %{side}_win_pct_{W}Winning percentage over the last W games
Run Diff{side}_run_diff_{W}Average run differential per game over W games
Pythagorean{side}_pythag_{W}Expected win% from runs scored/allowed over W games

The 7 and 14-game windows capture very recent hot/cold streaks; the 60-game window reflects long-term team quality. All use shift(1) to prevent leakage — the current game's result is never included.

Additional rolling features include:

  • EWMA (span=20): exponentially weighted versions of win%, run diff, and Pythagorean that weight recent games more heavily
  • Home/away splits: separate rolling win% and Pythagorean for each team's home-game and away-game subsequences
  • Win/loss streak: consecutive wins (positive) or losses (negative)
  • Rest days: calendar days since last game (capped at 10)
  • Run std (30-game): scoring variance — high-variance offences are less predictable
  • One-run game win% (30-game): close-game resilience

Pitcher Quality (8+ features)

Starting pitcher quality is captured through prior-season stats from the MLB Stats API and in-season cumulative stats from Retrosheet gamelogs.

  • Prior-season ERA, K/9, BB/9, WHIP for the home and away starters
  • In-season ERA blend: a Bayesian shrinkage estimator that blends the pitcher's current-season gamelog ERA with the league average (ERA=4.50), weighted by the number of starts. This prevents extreme ERA values from small samples from dominating.
  • SP estimated wOBA allowed (Statcast): how hard the opposing starter was hit in the prior season, based on exit velocity and launch angle data

Advanced Team Metrics (26+ features)

From FanGraphs (prior season, available 2002+):

SideBattingPitching
Home & AwaywOBA, Barrel%, Hard Hit%, ISO, BABIP, xwOBAFIP, xFIP, K%, BB%, HR/FB, WHIP

Differential features are also computed: Pythagorean diff (30-game and EWMA), home/away split diff, SP ERA diff, wOBA diff, FIP diff, xwOBA diff, WHIP diff, and ISO diff. These directly encode the matchup advantage/disadvantage.

Context & Venue (5 features)

  • Park run factor — runs at this venue vs. league average (requires 30+ historical games)
  • Season progress — 0.0 at the opener, 1.0 on the final day; captures the effect of roster expansion and playoff pressure
  • Day/night flag — some teams and pitchers perform differently in day games
  • Interleague flag — interleague games (AL vs. NL) have historically different dynamics
  • Day of week — encodes travel and scheduling patterns (Monday getaway days, Sunday day games)

Vegas & Weather (5 features)

  • Vegas implied home win probability — the market's estimate, incorporating injury news, public money, and factors models may miss
  • Line movement — how much the line shifted from open to close, indicating sharp money or late-breaking news
  • Temperature (°F), wind speed (mph), humidity (%) — environmental conditions that affect ball flight and pitcher grip

Models

The system trains six models, each with different inductive biases. By combining them in a stacked ensemble, the system exploits their complementary strengths while minimising their individual weaknesses.

What is a model? A machine learning model is a mathematical function that takes pre-game features as input (Elo ratings, pitcher stats, rolling stats, etc.) and outputs a number between 0 and 1 representing the estimated probability that the home team wins.

Logistic Regression Baseline

Logistic regression is the simplest model in the system. It fits a linear equation to the features and passes the result through a sigmoid function to produce a probability. Because the decision boundary is a hyperplane, it captures additive effects — for example, a larger Elo differential increases home-win probability by a fixed amount regardless of other features.

All features are z-score standardised (subtract mean, divide by standard deviation) before fitting so that features on different scales contribute proportionally. Regularisation (L2/ridge, C=1.0) prevents overfitting by penalising very large coefficients.

Strengths: Fast, stable, interpretable. Feature importance is directly readable from the coefficients.
Limitations: Cannot capture interactions (e.g., "a good pitcher matters more in a pitcher's park") without explicit interaction features.

LightGBM Gradient Boosted Trees

LightGBM (Microsoft) grows an ensemble of shallow decision trees in sequence. Each tree corrects the residual errors of the previous ones. Unlike logistic regression, trees automatically capture non-linear interactions — for example, a high Elo differential combined with a good home/away split may carry a larger joint effect.

LightGBM uses a leaf-wise growth strategy (growing the leaf with the largest loss reduction first) and histogram-based binning for fast training. Hyperparameters are optimised via Optuna Bayesian search (200 trials). Typical optimal values: num_leaves≈63, learning_rate≈0.05, n_estimators≈500.

XGBoost Gradient Boosted Trees

XGBoost (DMLC) is another gradient-boosted tree library with a different regularisation scheme: it uses min_child_weight and separate L1/L2 penalties on leaf weights. Its predictions are complementary to LightGBM — they tend to disagree most on uncertain games near 50%, making them valuable ensemble partners.

XGBoost typically achieves the best single-model Brier score in this system. Hyperparameters are also Optuna-tuned. Typical: max_depth≈6, learning_rate≈0.05, n_estimators≈500.

CatBoost Gradient Boosted Trees

CatBoost (Yandex) uses ordered boosting and symmetric (oblivious) decision trees. Ordered boosting trains each tree on a random permutation of the data, reducing prediction shift (a form of target leakage in traditional boosting). Symmetric trees — where every node at the same depth uses the same split condition — tend to generalise well on tabular data and produce low-variance predictions.

CatBoost provides a structurally different third tree model for the ensemble. Its hyperparameters (depth, iterations, L2 leaf reg) are Optuna-tuned.

Neural Network (MLP) Deep Learning

A multi-layer perceptron (MLP) is a feed-forward neural network. The system uses three hidden layers (128 → 64 → 32 neurons) with ReLU activation functions and an Adam optimiser. Features are z-score normalised before training.

Neural networks capture non-linear patterns that are fundamentally different from tree-based splits. Their error surface is shaped differently, providing valuable model diversity for stacking. Regularisation is applied via L2 weight decay (alpha=0.001).

Stacked Ensemble Production Default

The stacked ensemble is the default production model and never sees raw features. Instead, it takes the calibrated probability outputs of all five base models as its five inputs and trains a logistic regression meta-learner to find the optimal blend:

Logistic prob ─┐ LightGBM prob ─┤ XGBoost prob ─┼──▶ Logistic meta-learner (C=0.5) ──▶ P(home win) CatBoost prob ─┤ MLP prob ─┘

Because each base model makes different errors, the meta-learner learns to up-weight whichever model is most confident in each probability range. The meta-learner is fit on the same held-out calibration split used for Platt scaling, so base-model probabilities are out-of-sample relative to the meta-learner. The slight regularisation (C=0.5) prevents overfitting to the calibration set.

Training Pipeline

Expanding-Window Cross-Validation

The system uses an expanding-window protocol to ensure all evaluation is fully out-of-sample and respects the temporal ordering of baseball seasons:

Season: 2000 2001 2002 2003 2004 ... 2024 2025 ├────────────────────────────────────────────────┤ Fold 1: [Train: 2000-2001] [Eval: 2002] Fold 2: [Train: 2000-2002] ........ [Eval: 2003] Fold 3: [Train: 2000-2003] ............ [Eval: 2004] ... Fold 24: [Train: 2000-2024] ............................ [Eval: 2025]

For each fold, the training data is further split: the last 20% of training rows (by time) are held out as a calibration set for Platt or isotonic calibration. The model never sees the evaluation season during training or calibration.

Probability Calibration

Why calibrate? A raw model might output "70%" but actually be correct only 60% of the time at that confidence level. Calibration adjusts the probabilities so that when the model says 70%, games actually have a ~70% home win rate.

The system supports two calibration methods:

Platt Calibration (Sigmoid)

Fits a logistic regression on the base model's log-odds output: P(win) = 1 / (1 + exp(-A × logit(raw) - B)). Best suited for models whose errors are roughly sigmoid-shaped (logistic regression, MLP). Uses a held-out calibration set to fit A and B.

Isotonic Calibration (Non-Parametric)

Fits a piecewise-constant, monotonically increasing function from raw probabilities to calibrated probabilities. Makes no assumptions about the shape of the calibration curve, so it works better for tree models (LightGBM, XGBoost, CatBoost) whose probability surfaces can have arbitrary shapes. Requires at least 500 calibration samples to avoid overfitting.

The system automatically applies isotonic calibration for tree models and Platt calibration for logistic regression and MLP.

Time-Weighted Training

Baseball evolves: the 2023 shift ban, pitch clock, and larger bases changed the game significantly. To adapt, the system applies exponential decay sample weights with a rate of 0.12 per season: 

weight(season) = exp(-0.12 × (max_season - season))

Example (training for 2025 evaluation):
  2024 weight = 1.00    2023 weight = 0.89
  2022 weight = 0.79    2020 weight = 0.55
  2015 weight = 0.30    2010 weight = 0.17
  2005 weight = 0.09    2000 weight = 0.05

This ensures the model adapts to recent rule changes and playing styles while still benefiting from the large historical sample.

Dynamic Feature Selection

Not all seasons have the same features available — short-window rolling stats (7 and 14-game) were added later and are not present in older feature files. The training pipeline automatically detects the intersection of available features across all season DataFrames and trains using only those features. This ensures:

  • Training never fails due to missing columns in older seasons
  • Future feature additions do not break the pipeline
  • The actual feature list used is stored in model metadata for reproducibility

Additionally, 12 redundant features (7 and 14-day variants that are highly correlated with the 15-day versions) are automatically pruned to reduce noise for tree models.

Evaluation & Metrics

All metrics are computed on fully out-of-sample data — the model has never seen the evaluation season during training or calibration.

Brier Score

The Brier score measures the mean squared error of probability predictions: 

Brier = (1/N) × Σ (predicted_prob - actual_outcome)²

where actual_outcome is 1 (home win) or 0 (away win). A perfect predictor scores 0.0; always predicting 0.5 scores 0.25. In baseball, with an inherent randomness floor, scores around 0.24 are near the theoretical limit for pre-game predictions.

Log Loss

Log loss penalises confident wrong predictions more severely than Brier score: 

LogLoss = -(1/N) × Σ [y × log(p) + (1-y) × log(1-p)]

A confident 95% prediction on a game that is lost incurs a much larger log loss penalty than a 55% prediction on the same game. This makes log loss more sensitive to overconfidence.

Accuracy

The simplest metric: what percentage of games did the model correctly predict the winner? The model "predicts" the home team if P(home win) ≥ 0.5. Baseball's inherent randomness limits accuracy to roughly 56–59% even for an excellent model. For comparison, always picking the home team yields about 54%.

Calibration Error

Games are grouped into 10 probability bins (quantiles). For each bin, the mean predicted probability is compared to the actual win rate. The calibration error is the average absolute difference across bins. A well-calibrated model has an error near 0 — meaning that among games where it predicts 70%, approximately 70% are actually won.

Predictions & Drift Monitoring

Prediction Snapshots

Every time predictions are generated, they are saved as an immutable snapshot in Parquet format at: 

data/processed/predictions/season=YYYY/snapshots/run_ts=<ISO>_<model_type>.parquet

Each snapshot records:

  • game_pk, home_team, away_team — game identity
  • predicted_home_win_prob — the model's probability estimate
  • run_ts_utc — when the prediction was generated (auto-set, immutable)
  • model_version, git_commit — exact code and model version
  • schedule_hash, feature_hash — SHA256 checksums of input data for provenance

Snapshots are never modified or deleted. This enables longitudinal study of how predictions change as the season unfolds and as models are retrained.

Drift Monitoring

After each prediction run, the system computes drift metrics by comparing the current snapshot to previous ones:

  • Incremental diff: current vs. the immediately prior snapshot — detects sudden shifts from data updates or retraining
  • Baseline diff: current vs. the first snapshot of the season — tracks cumulative drift over time

Each diff produces per-game deltas and aggregate metrics:

MetricDescription
mean_abs_deltaAverage absolute probability change across all games
p95_abs_delta95th percentile of absolute changes
max_abs_deltaLargest single-game probability change
pct_gt_0p01% of games with delta > 1 percentage point
pct_gt_0p05% of games with delta > 5 percentage points

Error Handling

The system defines a structured error taxonomy so that every failure is classified and traceable. All domain errors inherit from WinProbError:

Error ClassWhen It Fires
IngestionErrorRaw data download or parsing fails (e.g., Retrosheet unavailable, malformed API response)
APIErrorMLB Stats API communication failure (timeouts, rate limits, unexpected status codes)
CoverageErrorCrosswalk coverage falls below the 99% threshold
SchemaErrorData doesn't match expected schema (missing columns, wrong types)
DriftComputationErrorSnapshot diff or metrics aggregation fails
SnapshotIntegrityErrorImmutable prediction snapshot is corrupt or violates its schema

Silent failure is forbidden. Every except block in the codebase logs the error with context. Feature computation failures gracefully degrade to league-average defaults while logging a warning.

The web API includes a global exception handler that catches unhandled errors, logs the full traceback, and returns a safe JSON error response to the client.

The system is designed for durability and auditability. All raw data is preserved alongside processed outputs. Model artifacts include full metadata (training seasons, hyperparameters, feature list, checksums). Prediction snapshots are immutable and support longitudinal study across multiple seasons of model evolution.

Provenance guarantee: Every prediction can be traced back through the model artifact (which records its training seasons and feature version), to the feature files (which record their input data checksums), to the raw data files (which record their source URLs and SHA256 hashes).
